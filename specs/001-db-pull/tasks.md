# Tasks: `supalite db pull` — Baseline Schema Dump (001)

**Input**: [spec.md](spec.md), [plan.md](plan.md), [contracts/cli-contract.md](contracts/cli-contract.md), [quickstart.md](quickstart.md), [research.md](research.md)
**Approach**: live-Postgres TDD (repo convention — tests hit `DB_CONNECTION`); commit after every task.

## User Stories

- **US1 (P1)**: As a developer, I get a dependency-ordered, idempotent
  baseline SQL from a live database that round-trips onto an empty DB and
  re-applies to the source DB with zero errors (spec scenarios 1–3, 8;
  SC-001/002/004).
- **US2 (P2)**: As a developer, I control safety filters — extension-owned
  objects excluded by default with an opt-in include flag, and a plain-DDL
  mode (spec scenarios 4, 7; SC-003; FR-007/008).
- **US3 (P3)**: As a developer, I invoke everything through the
  `supalite db pull` CLI with `gen types`-consistent flags, default out path,
  stdout mode, and a clear diff-mode error (spec scenarios 1, 5, 6; SC-005;
  FR-001/002/017).

## Implementation strategy

- **MVP = US1** (T001–T011): after T011 the library API delivers the full
  guarantee (round-trip + idempotency) without a CLI.
- Incremental delivery: US2 hardens defaults/modes; US3 exposes the CLI; the
  polish phase makes it discoverable.
- Commit after every task (`feat(db-pull): ...` / `test(db-pull): ...` /
  `docs: ...`), keeping `npm test -- db-pull` green at each step.
- Every task's test commands run against live Postgres via `DB_CONNECTION`
  (default `postgresql://testuser:testpassword@localhost:5432/testdb`).
