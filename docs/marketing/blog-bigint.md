PostgreSQL's `BIGINT` (`int8`) goes up to **9,223,372,036,854,775,807**.
JavaScript's `Number.MAX_SAFE_INTEGER` is **9,007,199,254,740,991** (2⁵³−1).
Every integer above that is stored as the nearest representable double — so the
moment a real 64-bit ID crosses into your JS runtime, it can change value
without a single error being thrown.

This isn't theoretical. If you have `bigint` primary keys, Snowflake-style IDs,
Kakao/LINE/Discord user IDs, financial amounts in the smallest unit, or anything
generated by a sequence that has run long enough, you are in range.

So: given the exact column value `9223372036854775807`, what does each popular
client hand your code? I created a row, queried it four ways, and printed the
result, its JS type, and what `JSON.stringify` does with it.

## supabase-js — silently wrong

```js
const { data } = await supabase.from('t').select('big').eq('id', 1).single();
data.big;                 // 9223372036854776000   (number)
JSON.stringify(data.big); // "9223372036854776000"
```

PostgREST serializes `int8` as a JSON number, and `JSON.parse` rounds it. No
error, no warning. Your `...807` is now `...000`. This is the worst failure
mode because nothing tells you it happened.

## Prisma — precise, but explodes on serialize

```js
const row = await prisma.t.findUnique({ where: { id: 1 }, select: { big: true } });
row.big;                 // 9223372036854775807n  (BigInt)
JSON.stringify(row.big); // TypeError: Do not know how to serialize a BigInt
```

Prisma keeps the value (good) by returning a native `BigInt`. But the first time
that object hits `res.json(...)` or any `JSON.stringify`, it throws. You end up
sprinkling `BigInt.prototype.toJSON` monkey-patches or per-field `.toString()`
across your codebase.

## Drizzle — pick your poison

Drizzle makes you choose the mode per column:

```ts
bigint('big', { mode: 'number' }) // 9223372036854776000  -> precision lost
bigint('big', { mode: 'bigint' }) // 9223372036854775807n -> JSON.stringify throws
```

`mode: 'number'` is the tempting default ("it's just an int"), and it's the same
silent corruption as supabase-js. `mode: 'bigint'` is correct but inherits
Prisma's serialization problem. There's no built-in JSON-safe middle ground.

## The fix: number when safe, string when not

The pragmatic answer is to return a plain `number` while the value is inside the
safe range and fall back to a `string` when it would lose precision. You keep
ergonomic numbers for the 99% case, never lose precision, and `JSON.stringify`
always works.

That's the default in [SupaLite](https://github.com/genideas-labs/supalite), a
lightweight TypeScript Postgres client with a Supabase-style query builder:

```js
const { data } = await client.from('t').select('big').eq('id', 1).single();
data.big;                 // "9223372036854775807"  (string)
JSON.stringify(data.big); // "\"9223372036854775807\""  ✓ precise & JSON-safe
```

`bigintTransform` controls it and defaults to `'number-or-string'`. You can also
force `'number'`, `'string'`, or `'bigint'` if you have a specific need.

## Summary

| client | returns | `JSON.stringify` | result |
|---|---|---|---|
| supabase-js | `9223372036854776000` (number) | serializes wrong | ✗ silently corrupted |
| Prisma | `…807n` (BigInt) | throws | ✗ not JSON-safe |
| Drizzle | number → wrong, or BigInt → throws | — | ✗ pick your poison |
| SupaLite | `"9223372036854775807"` (string) | works | ✓ precise & JSON-safe |

Whatever client you use, the rule is the same: **don't let a 64-bit integer
become a JS `number` unless you've proven it's under 2⁵³.** A string at the
boundary is almost always what you actually want over the wire.
