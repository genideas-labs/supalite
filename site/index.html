Reads like Supabase

Keep the query ergonomics you already know. Here's the same query — fetch active users, newest first, page 2 (10 per page) — in each tool.

typescript
const { data } = await client
  .from('users')
  .select('id, name, email, created_at')
  .eq('status', 'active')
  .order('created_at', { ascending: false })
  .limit(10)
  .offset((page - 1) * 10);
typescript
const data = await prisma.user.findMany({
  select: { id: true, name: true, email: true, created_at: true },
  where: { status: 'active' },
  orderBy: { created_at: 'desc' },
  take: 10,
  skip: (page - 1) * 10,
});
typescript
const data = await db
  .select({ id: users.id, name: users.name, email: users.email, created_at: users.createdAt })
  .from(users)
  .where(eq(users.status, 'active'))
  .orderBy(desc(users.createdAt))

How much faster?

Same database, same query, four clients. supabase-js pays the REST/PostgREST tax; the direct drivers don't — and supalite keeps Supabase's API while matching them. Median (p50) latency, lower is better:

supabase-js REST1.00 ms
supalite direct0.24 ms 4.2×
Drizzle direct0.28 ms 3.6×
Prisma direct0.31 ms 3.2×

Methodology: local Supabase stack (PostgREST) vs direct Postgres, same machine, identical query (5 columns, limit(20)), 300 interleaved iterations after warmup. This isolates the HTTP/PostgREST API-layer overhead — real-world cloud deployments add network hops and cold starts on top, so the gap only widens. supalite, Drizzle, and Prisma all talk directly to Postgres, so they land in the same tier. Reproduce it in your own environment with benchmarks/ in the repo.

And your 64-bit IDs survive

PostgreSQL BIGINT reaches 9,223,372,036,854,775,807 — past JavaScript's safe integer limit (253-1). Here's what each client actually returns for that exact value:

client value returned in JS result
supabase-js 9223372036854776000 (number) ✗ silently wrong
Prisma 9223372036854775807n (BigInt) ✗ JSON.stringify throws
Drizzle number → wrong, or BigInt → throws ✗ pick your poison
supalite "9223372036854775807" (string) ✓ precise & JSON-safe

supalite's bigintTransform defaults to 'number-or-string': a plain number when the value fits safely, a string when it wouldn't — so it's precise and survives JSON.stringify with zero config. Configurable per client.

What's supported

Supported

  • Select, filters, order, pagination
  • PostgREST-style embeds (related(*), !inner)
  • Insert, update, delete, upsert (incl. ignoreDuplicates)
  • RPC (incl. single / maybeSingle)
  • Concurrency-safe transactions
  • Multi-schema, views, functions, enums

Not supported — use supabase-js

  • Auth
  • Storage
  • Realtime

SupaLite vs Prisma / Drizzle

Choose SupaLite when you want a thin, SQL-close query layer — especially when migrating off Supabase — and handle migrations separately. Choose Prisma or Drizzle when you want schema-first modeling and built-in migrations.

Quickstart

  1. 1Install

    bash
    npm install supalite

  2. 2Connect

    typescript
    import { SupaLitePG } from 'supalite';
import { Database } from './types/database';

const client = new SupaLitePG<Database>({
  connectionString: process.env.DB_CONNECTION,
});

  3. 3Query & transact

    typescript
    const { data, error } = await client
  .from('users')
  .select('id, name, email')
  .eq('status', 'active')
  .order('created_at', { ascending: false })
  .limit(10);

// Transactions — not supported by the Supabase client
await client.transaction(async (tx) => {
  const { data: user } = await tx
    .from('users')
    .insert({ name: 'Hong', email: 'hong@example.com' })
    .select()
    .single();

  await tx
    .from('profiles')
    .insert({ user_id: user.id, bio: 'Transaction profile' });
});

  4. 4Generate types from your database

    bash
    npx supalite gen types \
  --db-url "postgresql://user:pass@localhost:5432/db" \
  --out database.types.ts

Frequently asked questions

What is SupaLite?
SupaLite is a lightweight TypeScript PostgreSQL client with a Supabase-style query builder. It keeps the familiar .from().select().eq() API but talks directly to Postgres, so you get the query builder, RPC, and transactions with less overhead than the supabase-js REST client. MIT licensed, on npm as supalite.
Is SupaLite faster than supabase-js?
Yes, for database queries. supabase-js goes through the REST/PostgREST HTTP layer; SupaLite connects directly to Postgres. In a same-machine benchmark (identical query, 300 iterations) SupaLite's p50 latency was ~0.24 ms vs ~1.0 ms for supabase-js — about 4× lower — putting it in the same tier as Drizzle and Prisma. Cloud deployments add network hops on top, widening the gap.
Does SupaLite support transactions?
Yes. SupaLite provides concurrency-safe transactions via client.transaction(async (tx) => { ... }), which the Supabase client does not support. Each call runs on its own pooled connection, commits on success, and rolls back on error.
How does SupaLite handle 64-bit integers (BIGINT)?
PostgreSQL BIGINT can exceed JavaScript's safe integer limit (253-1). SupaLite's bigintTransform defaults to 'number-or-string': a number when safe, a string when not — precise and JSON-serializable. supabase-js returns a JSON number and loses precision; Prisma and Drizzle return a BigInt that throws on JSON.stringify.
SupaLite vs Prisma vs Drizzle — which should I use?
SupaLite is a thin, SQL-close query layer with Supabase ergonomics — pick it when you want minimal abstraction or are migrating off Supabase and handle migrations separately. Pick Prisma or Drizzle when you want schema-first modeling and built-in migrations. All three connect directly to Postgres and perform similarly; SupaLite adds JSON-safe BIGINT handling by default.
Does SupaLite support auth, storage, or realtime?
No. SupaLite focuses on the database layer (query builder, RPC, transactions). For auth, storage, or realtime, keep using supabase-js or dedicated services (Cognito/Google Identity, S3/GCS, a pub/sub or LISTEN/NOTIFY gateway).
Can I migrate from Supabase to SupaLite?
Yes. SupaLite mirrors the Supabase query-builder API, so most .from().select() queries work unchanged. It replaces only the DB query layer — you'll still need alternatives for auth/storage/realtime. It also includes supalite gen types for Supabase-compatible type generation.
How do I install SupaLite?
Run npm install supalite. It's an MIT-licensed npm package that works with any PostgreSQL database. Connect with new SupaLitePG({ connectionString }) and query with the Supabase-style builder.
